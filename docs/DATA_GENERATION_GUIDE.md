# Data Generation Guide — New Class / Subject

How to extract NCERT (or similar) PDFs for a **new class or subject** and load them into PostgreSQL with the same structure as existing books.

---

## Overview

```
PDF upload → Pipeline steps 1–10 → step10_canonical.json → PostgreSQL
```

Every book follows the same hierarchy and ID rules. The API (`/api/courses`, MCQ layer, search) assumes this structure — do not invent alternate shapes.

---

## Step 1 — Register the book

Upload the PDF with correct metadata. These fields become `books.title`, `books.subject`, and `books.class_level` in PostgreSQL.

### Via UI

1. Open the frontend → upload PDF.
2. Set:
   - **Title:** `NCERT Class 9 Geography` (descriptive, includes class)
   - **Subject:** `Geography` (used for course listing)
   - **Class level:** `9` (string — matches exam app filters)

### Via API

```bash
curl -X POST http://localhost:8000/api/books/upload \
  -F "file=@NCERT-Class-9-Geography.pdf" \
  -F "title=NCERT Class 9 Geography" \
  -F "subject=Geography" \
  -F "class_level=9"
```

Save the returned `book.id` (UUID) — this is the `book_id` everywhere.

**Naming conventions**

| Field | Example | Notes |
|-------|---------|-------|
| `title` | `NCERT Class 10 History` | Full display name |
| `subject` | `History`, `Geography`, `Science` | One subject per book/PDF |
| `class_level` | `9`, `10`, `11`, `12` | String, not integer |

One PDF = one book = one `book_id`. Do not mix subjects in a single upload.

---

## Step 2 — Run the extraction pipeline (steps 1–10)

Run steps **in order** for the new `book_id`. Each step reads the previous step's JSON from `pipeline_output/{book_id}/`.

| Step | Output file | Purpose |
|------|-------------|---------|
| 1 | `step01_pdf_reader.json` | Page text from PDF |
| 2 | `step02_layout_extraction.json` | Layout blocks + coordinates |
| 3 | `step03_content_blocks.json` | Merged content blocks |
| 4 | `step04_reading_order.json` | Reading order per page |
| 5 | `step05_block_classification.json` | Roles: paragraph, heading, figure, etc. |
| 6 | `step06_hierarchy.json` | Chapters → sections → subsections |
| 7 | `step07_paragraphs.json` | Merged paragraphs |
| 8 | `step08_image_table.json` | Figures, tables, activities, glossary |
| 9 | `step09_validation.json` | Cross-step validation (must pass) |
| 10 | `step10_canonical.json` | **Canonical store input** |

Use the UI pipeline runner or `POST /api/books/{book_id}/pipeline/run/{step}`.

**Step 9 must pass** before step 10. If validation fails, fix upstream steps — do not skip validation.

---

## Step 3 — Canonical JSON structure (`step10_canonical.json`)

This is the **only** file loaded into PostgreSQL. Shape is defined in `backend/pipeline/models.py` (`Step10CanonicalResult`).

### Top-level shape

```json
{
  "step": 10,
  "step_name": "canonical_json",
  "book": { "book_id", "title", "subject", "class_level", "filename", "total_pages" },
  "chapters": [ ... ],
  "sections": [ ... ],
  "subsections": [ ... ],
  "blocks": [ ... ],
  "paragraphs": [ ... ],
  "figures": [ ... ],
  "activities": [ ... ],
  "tables": [ ... ],
  "glossary": [ ... ],
  "counts": { "chapters": 8, "paragraphs": 513, ... },
  "validation": { "structure_complete": true, "errors": [] }
}
```

### Hierarchy (must stay consistent)

```
book (book_id)
 └── chapter (chapter_id)           e.g. CH_I, CH_II
      └── section / topic           e.g. CH_I_TOPIC_1  (is_overview for topic 0)
           └── subsection           e.g. CH_I_TOPIC_1_SUB_001
                └── content:
                     blocks, paragraphs, figures, activities, tables, glossary
```

Every content row carries `book_id` plus optional `chapter_id`, `section_id`, `subsection_id` for scoping.

---

## Step 4 — ID conventions (do not change)

IDs are generated by the pipeline and used as **primary keys** in PostgreSQL. Downstream systems (MCQ bank, user progress) depend on stability across reloads.

| Entity | Pattern | Example |
|--------|---------|---------|
| Book | UUID from upload | `3e0531a5-e7c1-4995-bdb3-979ad5ef4983` |
| Chapter | `CH_{ROMAN}` | `CH_I`, `CH_XII` |
| Section (topic) | `{chapter_id}_TOPIC_{n}` | `CH_II_TOPIC_3` |
| Overview section | `{chapter_id}_TOPIC_0` | `CH_I_TOPIC_0` (`is_overview: true`) |
| Subsection | `{section_id}_SUB_{nnn}` | `CH_I_TOPIC_2_SUB_001` |
| Block | `CB{6 digits}` | `CB000661` |
| Paragraph | `P{5 digits}` | `P00042` |
| Figure | `FIG{5 digits}` | `FIG00003` |
| Activity | `ACT{5 digits}` | `ACT00012` |
| Table | `TBL{5 digits}` | `TBL00001` |
| Glossary entry | `GE{hash}` | auto-generated on DB load |

**Rules**

- Never hand-edit IDs after first load if MCQs or user data reference them.
- Reloading a book **deletes and replaces** all rows for that `book_id` (cascade).
- `section_ids` on each chapter must match actual `sections` entries.
- `paragraph_ids` on sections/subsections must reference real `paragraphs`.

## Step 5 — Load into PostgreSQL

```bash
# Migrations (once per environment)
alembic upgrade head

# Load one book
python -m backend.db.load_canonical <book_id>

# Or via API
curl -X POST http://localhost:8000/api/books/<book_id>/canonical/load
```

### Verify

```bash
./scripts/db_tables.sh
curl http://localhost:8000/api/books/<book_id>/canonical/status
curl http://localhost:8000/api/admin/db/tables
```

Expected `canonical/status` response includes row counts and `validation_passed: true`.

### PostgreSQL tables populated

| Table | Source in step10 |
|-------|------------------|
| `books` | `book` + `counts` + `validation` |
| `chapters` | `chapters[]` |
| `sections` | `sections[]` |
| `subsections` | `subsections[]` |
| `blocks` | `blocks[]` |
| `paragraphs` | `paragraphs[]` |
| `figures` | `figures[]` |
| `activities` | `activities[]` |
| `tables` + `table_cells` | `tables[]` |
| `glossary_entries` | `glossary[]` |

See [CANONICAL_STORE.md](./CANONICAL_STORE.md) for API query examples.

## Step 6 — Checklist before marking a book "done"

- [ ] PDF is official NCERT (or agreed source) for the target class/subject
- [ ] `title`, `subject`, `class_level` set correctly at upload
- [ ] Pipeline steps 1–9 all complete; step 9 validation **passed**
- [ ] `step10_canonical.json` exists under `pipeline_output/{book_id}/`
- [ ] `validation.structure_complete` is `true` in step10
- [ ] `counts` look reasonable (chapters match PDF, paragraphs > 0)
- [ ] `canonical/load` succeeded; `canonical/status` shows expected counts
- [ ] Spot-check in Swagger:
  - `GET /api/courses` — new subject appears
  - `GET /api/courses/{book_id}` — chapters and topics listed
  - `GET /api/courses/.../topics/{topic_id}/intro` — readable content

## Adding another class — quick recipe

Example: Class 9 Science after Class 10 History exists.

1. Upload `NCERT-Class-9-Science.pdf` with `subject=Science`, `class_level=9`.
2. Run pipeline steps 1–10 for the new `book_id`.
3. `python -m backend.db.load_canonical <new_book_id>`.
4. Confirm both books appear in `GET /api/courses`.

Each class/subject is an independent `book_id`. The same PostgreSQL database holds all books; the courses API groups by `subject` + `class_level`.

## What not to commit

Generated artifacts stay local (see `.gitignore`):

- `pipeline_output/` — step JSON including `step10_canonical.json`
- `uploads/` — PDF files
- `ocr_output/`, `json_output/`
- `exports/` — DB dumps for sharing
- `.env` — secrets and `DATABASE_URL`

Share canonical data with teammates via `pg_dump` / `exports/` or by re-running the pipeline on their machine.

## Troubleshooting

| Problem | Fix |
|---------|-----|
| Step 9 fails | Read `step09_validation.json` errors; re-run failed upstream step |
| `step10 not found` | Complete step 10 after step 9 passes |
| Wrong chapters/topics | Re-check step 6 hierarchy; NCERT PDF must have detectable chapter headings |
| Empty paragraphs | Re-run steps 5–7; check step 5 paragraph block classification |
| Load fails with FK errors | step10 IDs inconsistent — re-run step 10 from clean step 6–9 |
| Subject missing in UI | Run `canonical/load`; confirm `books` row exists |
